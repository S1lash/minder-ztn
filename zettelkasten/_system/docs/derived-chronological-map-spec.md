## 1. Problem

Hub `## Хронологическая карта` is currently **curator-edited** —
`/ztn:maintain` proposes additions; owner / earlier-skills curate the
actual map manually. Records carry `projects:` arrays; if a record's
primary project is hub-X, by intent it belongs in hub-X's map.

Drift observed in any sufficiently large base under ARCH-A semantics:
- records with `projects.length == 2` lacking boundary annotation
- occasional overcount records (3+ entries pre-tightening)
- mismatch between records' `projects:` array and hub chronological-map
  membership when the hub map is curator-edited

ARCH-A (forward-lint Scan A.8 + tightened `/ztn:process` prompt + a
trajectory-vs-project reclassification migration) **prevents** new
drift but doesn't **eliminate** it: any future curator omission, prompt
regression, or manual hub edit reintroduces mismatch.

## 2. Goal of ARCH-B

Make the hub `## Хронологическая карта` an **invariant-by-construction**
view, not a curator-edited section. After ARCH-B:

- Every record where `hub.id` ∈ record.projects appears in the hub's map
- Map ordering is deterministic (chronological by record.created)
- Editorial exclusions still possible — explicit, surfaced, audited

Mismatch becomes impossible (not «monitored»).

## 3. Architecture summary

### 3.1 Hub frontmatter — new fields

```yaml
hub_kind: project | trajectory | domain   # already added in ARCH-A
chronological_map_mode: derived | curated # NEW — default derived
excluded_from_map:                        # NEW — list of record IDs
  - {record-id}                            #   to exclude from derived map
  - ...
excluded_from_map_reasons:                # NEW — same length as above
  - {one-line reason for exclusion}        #   audit trail
```

`chronological_map_mode: curated` — backward-compat opt-out. Hubs that
need full curator control (e.g., narrative-hubs, special editorial
choices) keep the old behaviour. Default for project-hubs is `derived`.

### 3.2 Body section — replaced

Current:
```markdown
## Хронологическая карта

| Дата | Заметка | Тип | Что произошло |
|------|---------|-----|---------------|
| 2026-03-13 | [[note-id\|title]] | reflection | ... |
| ...        |                   |             | ... |
```

ARCH-B (when `chronological_map_mode: derived`):
```markdown
<!-- AUTO-GENERATED by /ztn:maintain — do not hand-edit between markers -->
## Хронологическая карта (derived)

| Дата | Заметка | Тип | Что произошло |
|------|---------|-----|---------------|
{rows derived from records' frontmatter — chronological by record.created}

### Excluded from map (editorial)

| Record | Reason |
|--------|--------|
{rows from excluded_from_map[*] + excluded_from_map_reasons[*]}

<!-- /AUTO-GENERATED -->
```

The auto-region is delimited by HTML comments. Hand-edits between
markers are silently overwritten on next `/ztn:maintain` run (event
emitted to log).

«Что произошло» column derivation: pull from record's
`description:` frontmatter field (1-line). If absent → derive from
title; if title also absent → use record-id verbatim.

«Тип» column derivation: pull from record's `kind:` (records) or
`types:[0]` (knowledge notes). Map to display name per
`5_meta/PROCESSING_PRINCIPLES.md` Note Types reference.

### 3.3 /ztn:maintain — new step

After existing hub maintenance steps, add **Step M.X: Derived Map
Render**.

Pseudocode:
```python
for hub in glob("5_meta/mocs/hub-*.md"):
    if hub.frontmatter.chronological_map_mode != "derived":
        continue
    project_id = hub.id.removeprefix("hub-")
    excluded = set(hub.frontmatter.excluded_from_map or [])
    members = [
        note for note in iter_all_notes()
        if project_id in (note.frontmatter.projects or [])
        and note.id not in excluded
    ]
    members.sort(key=lambda n: n.frontmatter.created)
    rows = [
        render_row(date=n.created, link=n.id,
                   kind=n.kind or n.types[0],
                   summary=n.description or n.title or n.id)
        for n in members
    ]
    body = re.sub(
        r"<!-- AUTO-GENERATED.*?<!-- /AUTO-GENERATED -->",
        render_auto_block(rows, excluded),
        hub.body,
        flags=re.DOTALL,
    )
    write(hub, body)
```

### 3.4 /ztn:lint — Scan A.8 update

After ARCH-B ships, the `projects-array-unknown-id` check tightens:
project ID must resolve to a hub with `hub_kind: project` AND
`chronological_map_mode: derived` for the auto-completeness invariant
to apply. Hubs with `chronological_map_mode: curated` are flagged for
explicit owner audit when they accumulate inbound records (new scan
sub-rule: «curated hub with > N inbound primary records — review»).

`projects-array-non-project-hub` and `projects-array-overcount` rules
unchanged from ARCH-A.

### 3.5 Manifest impact

`batch-format.md` §«hubs» section gets two new fields per hub-state:

```json
{

  "hub_kind": "project",
  "chronological_map_mode": "derived",

  ...
}
```

Format version bump: 2.0 → 2.1 (minor — additive only). Minder ingest
worker accepts unknown-minor; no breakage.

## 4. Migration plan

### 4.1 Pre-flight

- All project-hubs must carry `hub_kind: project` (set explicitly even
  though absent → default `project` — to surface intent).
- All trajectory + domain hubs must carry `hub_kind: trajectory` /
  `hub_kind: domain`. ARCH-A Phase 2 sets it for any hub the owner
  reclassified out of project status; remaining hubs need an audit pass.

### 4.2 Hub kind audit (one-shot script)

`_system/scripts/migrations/2026-05-XX-hub-kind-classification.py`:
- Iterate `5_meta/mocs/hub-*.md`
- For each — propose `hub_kind` based on signals:
  - hub with `domains:` ⊃ {personal arc, career, learning} → trajectory
  - hub matching a row in `1_projects/PROJECTS.md` → project
  - else → domain (with surface-back to owner for confirmation)
- Surface as CLARIFICATION batch; owner classifies, then re-run with
  `--apply`.

### 4.3 Body migration (one-shot script)

`_system/scripts/migrations/2026-05-XX-derive-hub-maps.py`:
- For each hub with `chronological_map_mode: derived` (default after
  step 4.2)
- Render new auto-block from current records' projects:
- Compare to existing body section — emit diff per hub
- Surface diff batch to owner via CLARIFICATION
  `derived-map-diff-vs-curated`
- After owner approves batch, write auto-block to hub bodies

### 4.4 Excluded-from-map seeding

For records where the existing curated map omitted them but record's
projects: includes the hub:
- Add to `excluded_from_map:` with a placeholder reason (`legacy curator
  exclusion — owner audit pending`)
- Owner reviews placeholder reasons in a follow-up resolve round; either
  confirms exclusion with proper reason OR removes from list (record
  joins the map)

### 4.5 Acceptance criteria

ARCH-B is done when:
- [ ] `/ztn:maintain` Step M.X implemented + tests pass
- [ ] All ~30 hubs migrated (kind + chronological_map_mode set)
- [ ] All record→hub mismatches resolve to one of: in map (auto) OR in
      excluded_from_map (with reason)
- [ ] Lint Scan A.8 reports zero `projects-array-non-project-hub` and
      zero `projects-array-overcount` events
- [ ] Manifest format-version bumped, Minder ingest passes integration
      test on derived-map fields
- [ ] One round of `/ztn:maintain` + `/ztn:lint` runs clean (no new
      CLARIFICATIONs from this scan family)

## 5. Risks

| Risk | Mitigation |
|---|---|
| Auto-overwrite destroys curator edits | HTML-comment markers; only region inside markers is overwritten; markers are searched, not assumed; if missing → skip and log warning |
| Order changes after derive (if record.created varies) | Sort key is `created`; no future writes to `created` after note creation; stable |
| Records drift back into hub via /ztn:process re-runs | Forward-lint Scan A.8 is the gate; Phase 1c shipped catches it pre-merge |
| 30-hub migration introduces bugs | Each hub processed atomically; per-hub diff surfaced before write; rollback via git |
| Records w/o `description:` frontmatter | Spec degrades to title or record-id; cosmetic only |
| `excluded_from_map_reasons` desync with `excluded_from_map` | Lint scan A.X.Y: arrays must be same length; otherwise CLARIFICATION |

## 6. Test plan

- Unit: `_system/scripts/tests/test_derived_map.py`
  - Single hub, 5 records, default order
  - Hub with 2 excluded records — output skips them, exclusion table renders
  - Hub with `chronological_map_mode: curated` — body untouched
  - Hub with malformed AUTO-GENERATED markers — error surfaced, no write
- Integration: full `/ztn:maintain` run on a fixture base; before/after
  diff stable across re-runs (idempotency)
- Migration script tests: dry-run vs apply; confirm no false positives;
  confirm placeholder reasons are inserted

## 7. Out of scope (defer)

- Per-domain map sub-sectioning (e.g., separating `architecture` vs
  `operations` sub-streams in a single project hub) — addressed in Phase 6 hub
  split assessment, separate decision.
- Hub merge / trajectory-hub formal definition — covered partially in
  ARCH-A (hub_kind), no further work in ARCH-B.
- Cross-hub link section (`## Связанные хабы`) auto-derivation — current
  manual `related_notes:` field is fine; not blocking ARCH-B.

## 9. Open questions for owner

These need owner answer BEFORE implementation:

1. **Default `chronological_map_mode` for new hubs**: `derived` (recommended)
   or explicit-opt-in `curated`? This spec assumes `derived` default; if
   owner prefers explicit, change Step 4.1 to surface every hub for
   classification.

2. **«Что происходит» column derivation hierarchy**: is `description:`
   the right primary source? Some records carry rich `description:`,
   others have `null` or `''`. Fallback to title — acceptable, but
   may produce uneven map quality across hubs.

3. **Exclusion default reason**: when seeding excluded_from_map_reasons
   during 4.4, use placeholder text or surface every record for
   per-record reason capture? Latter is more work but produces clean
   audit trail.

4. **Trajectory & domain hubs — keep chronological maps?** If yes,
   `chronological_map_mode: curated` stays; if no, drop the section
   entirely from those hub bodies. Recommend curated — trajectories
   often have narrative arcs that don't fit auto-derivation.
