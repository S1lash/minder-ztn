# Constitution — Protocol Document

> Curated typed set of axioms, principles, and rules that govern how Claude agents
> (Claude Code, `/ztn:*` pipelines, future MCP consumers) reason and act on owner's
> behalf. Lives as markdown + YAML frontmatter under `0_constitution/`. Single source
> of truth; every derivative view (harness rules, SOUL.Values, MCP payload, bootstrap
> export) is auto-generated from this tree.
>
> **This file is the protocol.** It defines schema, type hierarchy, conflict
> resolution, scope semantics, writeability, and lifecycle. Individual principles
> live as separate atomic notes under `axiom/`, `principle/`, `rule/`. Registry of
> active principles is auto-maintained in `_system/views/CONSTITUTION_INDEX.md`.

**Related documents:**
- [`_system/views/CONSTITUTION_INDEX.md`](../_system/views/CONSTITUTION_INDEX.md) — auto-generated registry of active principles
- [`_system/SOUL.md`](../_system/SOUL.md) — identity summary (Values zone auto-rendered from `core: true` notes)

## 1. Purpose & Position in the Platform

The constitution is the **rules-of-engagement layer** for any Claude agent acting on
owner's behalf. It answers three questions for any non-trivial decision:

1. *Does this align with a stated principle?* (alignment check)
2. *Does this violate a rule?* (hard boundary check)
3. *Is this a trade-off between principles?* (surfaces reasoning for logged decision)

**What it is not.** It is not:
- A knowledge base (that is ZTN `_records/` + PARA folders).
- A task/goal list (that is `_system/TASKS.md`).
- A people registry (that is `3_resources/people/PEOPLE.md`).
- A session journal (that is `_sources/` + `/ztn-recap`).

**Quality-first layer.** Unlike ZTN knowledge (capture-first: better save too much
than lose facts), constitution is curated narrowly. Noise in the constitution causes
agents to behave like someone who is not owner. Principles enter via narrow triggers
and human review; they never accumulate on autopilot.

**Kernel as discipline artifact.** The `core: true` set (target 5–8 notes, see §13
invariant #13) is not a "tier" — it is the irreducible nucleus from which the rest of
the tree reconstructs by derivation. Compression is a human act performed at curation
time; the infrastructure only guards against bloat (`core-bloat` CLARIFICATION when
count > 10) and surfaces stated-vs-actual drift (Scan F.2) on real decisions. The system
cannot manufacture essence — it holds the space in which discipline applies.

**Derived surfaces (all auto-generated, read-only for consumers):**

| Surface | Generated by | Consumer |
|---|---|---|
| `_system/views/constitution-core.md` (advisory ≤80 lines) | `gen_constitution_core.py` | Claude Code harness; symlink `~/.claude/rules/constitution-core.md` → this file once per machine |
| `_system/views/CONSTITUTION_INDEX.md` | `gen_constitution_index.py` | Human navigation + lint |
| `_system/SOUL.md` Values zone (between markers) | `render_soul_values.py` | `/ztn:process`, `/ztn:maintain`, `/ztn:lint` via SOUL load |

All three generators are orchestrated by `regen_all.py` (and the
`/ztn:regen-constitution` skill). Every pipeline that reads a derived view
regenerates first — single trigger, one consistent rule.

Shareable exports for external users (wife / friends) are not part of the
current surface set. Principles carry the `scope` field from day one so
filtering becomes a one-line change when sharing ships.

### Committed derived views — why

`_system/views/constitution-core.md` and `_system/views/CONSTITUTION_INDEX.md` are
committed to git, not regenerated on clone. Reason: the first pipeline
run after a fresh clone reads SOUL.md
(which imports the Values auto-zone); if the derived views were absent
on clone, SOUL would be stale until someone explicitly runs the regen
skill. Committing them turns cloning into «works immediately». Every
subsequent regen produces a small timestamp diff — that is expected;
see the `/ztn:regen-constitution` SKILL rollback section if you need to
discard accidental regens.

### Placeholder principles — schema-only fixture, not content

`status: placeholder` is a **reserved slot in the enum** for infrastructure
smoke-testing — nothing more. The active constitution contains **zero**
placeholder principles; everything under `0_constitution/` is real
content. If a future infra change (new scan, new pipeline) needs a
temporary fixture, a placeholder MAY be introduced transiently, but must
be removed before derived views export. Placeholders are **visible** in
`CONSTITUTION_INDEX.md` with the `PH` status flag if present, and **never**
reach any derived surface (invariant #18 below) — this is enforced by
`gen_constitution_core.py` and `render_soul_values.py` exclusion logic.

## 2. Type Hierarchy

Three types, ordered by abstraction:

```
    [axiom]       Fundamental truth. No obligation to derive from anything.
       │          priority_tier: 1 by default. Rare. Stable over years.
       │
       │ derived_from (optional)
       ▼
  [principle]     Behavioral pattern. Explains HOW to act.
       │          priority_tier: 1-3. May be binding: hard | soft.
       │
       │ derived_from (optional)
       ▼
     [rule]       Hard 1/0 constraint. No judgment. Binary.
                  priority_tier: 1 always. Rare. Example: "No work email after 22:00".
```

**Rule of thumb for picking type:**
- *"Why do I care about X?"* → `axiom` if the answer is "it just is, it's part of who I am".
- *"How do I act when X comes up?"* → `principle`.
- *"What is a line I never cross regardless of context?"* → `rule`.

**No other types.** If a candidate does not fit — it is probably a preference, a
heuristic without rule-content, or knowledge. It belongs elsewhere (ZTN notes,
project conventions, team rules).

## 3. Frontmatter Schema

Every file in `0_constitution/{axiom,principle,rule}/{domain}/*.md` has this
frontmatter. **Bold** fields are required.

```yaml
---
# Identity
id: axiom-identity-001                      # REQUIRED. {type}-{domain}-{NNN}
                                            # NNN is 3-digit monotonic per-type
                                            # per-domain sequence.
title: If it can be better, it should be    # REQUIRED. Short, behavioral form
                                            # where possible.
type: axiom                                 # REQUIRED. axiom | principle | rule
domain: identity                            # REQUIRED. See §5 Domains.

# Content
statement: >                                # REQUIRED. One sentence, behavioral.
  Choose the higher-quality path when
  the system is likely to outlive this
  decision.

# Classification
priority_tier: 1                            # REQUIRED. 1 | 2 | 3
framing: positive                           # positive (default) | negative
binding: hard                               # hard (default) | soft
core: true                                  # true | false. Target: 5–8 notes
                                            # total with core: true across whole
                                            # constitution. Only core=true feeds
                                            # SOUL.Values and harness core view.
                                            # Full tree accessible via
                                            # /check-decision.

# Access control — see §7
scope: shared                               # REQUIRED. shared | personal |
                                            # sensitive
applies_to: [claude-code, ztn, chatgpt]     # REQUIRED. Closed enum subset; see §7.

# Relationships
derived_from: []                            # list of {id}; empty for pure axioms
contradicts: []                             # list of {id}; explicit known conflicts

# Governance
confidence: proven                          # proven | working | experimental
status: active                              # active | candidate | archived |
                                            # placeholder
                                            # (placeholder = synthetic test data,
                                            # removed before content population.)

# Lifecycle
created: 2026-04-20                         # ISO date, set once.
last_reviewed: 2026-04-20                   # ISO date. Bumped manually on review.
last_applied: null                          # ISO date. Auto-bumped by
                                            # /check-decision on citation.
source_weight:
  own_experience: 5                         # 0–5 Dalio believability weight.
  external_author: 0                        # 0–5 external voice weight.
---
```

### Enum reference

| Field | Allowed values |
|---|---|
| `type` | `axiom`, `principle`, `rule` |
| `domain` | `identity`, `ethics`, `work`, `tech`, `relationships`, `health`, `money`, `time`, `learning`, `ai-interaction`, `meta` |
| `priority_tier` | `1`, `2`, `3` |
| `framing` | `positive`, `negative` |
| `binding` | `hard`, `soft` |
| `core` | `true`, `false` |
| `scope` | `shared`, `personal`, `sensitive` |
| `applies_to` | subset of `claude-code`, `ztn`, `chatgpt`, `claude-desktop`, `life-advice`, `work-code`, `bootstrap`, `minder` |
| `confidence` | `proven`, `working`, `experimental` |
| `status` | `active`, `candidate`, `archived`, `placeholder` |

**Closed enums.** Unknown enum values are a validation error. Extending an enum is
a schema change: add to this document first, then Python scripts pick it up on next
run. Consumers silently drop unknown values in `applies_to` lists (no crash).

### File naming

- Path: `0_constitution/{type}/{domain}/{slug}.md`
- Slug: lowercase kebab-case, matches id tail (`axiom-identity-001` → `001-if-it-can-be-better.md`
  or similar). Slug is descriptive; id is the authoritative key.
- One principle per file. No multi-principle files. No wildcards in imports.

## 4. Body Structure

Below the frontmatter, every note follows the same section order:

```markdown
# {Title}

## Statement
{Prose expansion of `statement`. Adds context, nuance the YAML one-liner cannot hold.}

## Why I hold this
{Lived reason. "howtothink.ai" prose-over-properties — what makes this true for owner.}

## Edge cases — when not to apply
{Dalio mental-map. Protects against dogmatism. Explicit anti-conditions.}

## Related
- [[principle/work/name-the-tradeoff]]
- [[rule/tech/no-silent-compromise]]

## Evidence Trail
- **YYYY-MM-DD** | {event} | [[source-or-record]] — {one-line}

## Source
extracted-from: {path or `hand-written` or `bootstrap-interview-{date}`}
```

**Order is fixed.** Scripts parse these sections by heading; reordering breaks
`render_soul_values.py` and `compact_evidence_trail.py`.

**Evidence Trail ordering.** Newest-first, consistent with Evidence Trail semantics
in knowledge notes.

## 5. Domains

Closed enum. Every principle carries exactly one domain. A principle that seems to
straddle two domains is usually either too broad (split it) or belongs in `meta`.

| Domain | What it covers |
|---|---|
| `identity` | Who I am. What I will not trade. Self-definition. |
| `ethics` | What is right / wrong independent of outcome. |
| `work` | How I approach professional execution, delivery, craft. |
| `tech` | Engineering judgment: architecture, tooling, code. |
| `relationships` | How I engage with partners, family, friends, colleagues. |
| `health` | Body, sleep, energy, longevity-related rules. |
| `money` | Financial judgment and personal finance principles. |
| `time` | Attention, cadence, how hours are spent. |
| `learning` | How I acquire and consolidate knowledge. |
| `ai-interaction` | Rules for how AI agents should behave on my behalf (meta counter-principles: "don't moralize on trivial decisions", etc.). |
| `meta` | Principles about the constitution system itself. |

New domains: add to this table first, then scripts and lint pick them up.

## 6. Priority & Conflict Resolution

### 6.1 Default tier mapping per domain

Any individual principle may override this; the table sets the sensible default when
a note is authored without an explicit tier decision.

| Domain | Default tier | Rationale |
|---|---|---|
| `ethics` | 1 | Non-negotiable by nature. |
| `identity` | 1 | Core of who I am. |
| `relationships` | 2 | Strong default; contextual exceptions exist. |
| `health` | 2 | Strong default; acute situations may override. |
| `ai-interaction` | 2 | Counter-principles need teeth to actually work. |
| `meta` | 2 | Strong default; rarely flexible. |
| `work` | 3 | Heuristic; trade-offs common. |
| `tech` | 3 | Heuristic; context-dependent. |
| `money` | 3 | Heuristic; life-stage-dependent. |
| `time` | 3 | Heuristic; cadence-dependent. |
| `learning` | 3 | Heuristic; goal-dependent. |

### 6.2 Resolution order

```
tier 1  ──beats──▶  tier 2  ──beats──▶  tier 3

Within the same tier:
  1. If either principle carries `contradicts: [other-id]` → the one with higher
     `confidence` wins (proven > working > experimental).
  2. Otherwise → /check-decision returns verdict = tradeoff; owner (or the agent
     routed to raise a CLARIFICATION) logs explicit reasoning.
```

**No separate CONFLICTS.md registry.** Conflicts are expressed as
`contradicts: [other-id]` in the frontmatter of each side. If the list of
real conflicts exceeds five, reconsider this choice.

### 6.3 Trade-off, not compromise

A trade-off is an **informed** choice between two valid principles. A compromise
is a **silent** degradation of a principle to avoid effort. The system accepts
trade-offs (logged with reasoning) and rejects silent compromises (global `HARD
RULE` per `~/.claude/CLAUDE.md`: "Any trade-off MUST be surfaced to the user
explicitly BEFORE implementing").

## 7. Scope Semantics & Access Control

### 7.1 The three scopes

`scope` is a property of the principle used as a **data marker for future
multi-user scenarios**. In the single-context dogfood phase all three
scopes are visible to every local consumer — no runtime narrowing is
applied. When multi-user scenarios land (sharing to wife / friends, MCP
tokens, public exports) the filter returns as a one-file change.

| Scope | Future-intended visibility | Current reality |
|---|---|---|
| `shared` | Safe to publish — everywhere, external AIs, bootstrap exports. | Visible (same as all). |
| `personal` | Only local agents of the owner. | Visible (same as all). |
| `sensitive` | Nothing auto-loads; only on explicit request. | Visible (same as all) in dogfood mode. Reconsidered when sharing ships. |

### 7.2 Consumer → visibility table (current)

Single-context dogfood: every local consumer sees every active,
non-placeholder, non-archived principle whose `applies_to` includes the
consumer tag. Future re-introduction of scope narrowing will fill this
table with per-consumer subsets.

| Consumer | Current | Future (multi-user) |
|---|---|---|
| Local Claude Code session | All scopes | shared + personal; sensitive on explicit request |
| `/ztn:process` / `/ztn:maintain` / `/ztn:lint` | All scopes via SOUL.Values + query | shared + personal |
| Claude Desktop (personal token) | n/a (dogfood offline) | shared + personal |
| ChatGPT via public MCP token | n/a | shared only |
| Bootstrap export to external user | n/a | shared only (physical strip) |

### 7.3 Enforcement layers

| Surface | Enforcement | Trust model |
|---|---|---|
| Local filesystem (current) | `gen_constitution_core.py` emits one `_system/views/constitution-core.md` with all scopes visible. Consumer filter (`applies_to` includes `claude-code`) is the only active narrowing. | Trusted local; single-user dogfood. |
| MCP server (future consumer) | Token → scope-set mapping; server filters **before** emission. Requests outside token scope → 403, not silent skip. | Untrusted client; server enforces. |
| Bootstrap export (future) | `export_for_sharing.py` physically strips non-shared files from the zip before delivery. | Trust applied at export time; nothing to leak later. |

**Fallback for missing env:** `env.CLAUDE_CONTEXT` unset → treat as `personal`
context (all three scopes visible). Single-user dogfood default; tightens to
`work` (shared-only) when work-repo `.claude/settings.json` sets the variable
explicitly. Multi-user scenarios (MCP with tokens, friend-clones) will invert
this default — each consumer picks their own context.

## 8. Writeability Matrix (L1–L4)

Four writeability levels, from most permissive to most restrictive.

| Level | Allowed actions | Who performs | Mechanism |
|---|---|---|---|
| **L1 — Autonomous** | Append to `## Evidence Trail`; bump `last_applied`; append to `_system/state/principle-candidates.jsonl`; regenerate `_system/views/CONSTITUTION_INDEX.md`; auto-render SOUL.Values zone. | Any pipeline or skill (`/ztn:process`, `/ztn:maintain`, `/ztn:lint`, `/check-decision`, `/ztn:capture-candidate`). | Write without commit. Dirty tree accumulates until owner commits. |
| **L2 — Auto with notify** | Merge exact-duplicate candidate into existing principle's Evidence Trail; add edge-case reference to `## Related`; mark candidate as suggested-noise for human review. | `/ztn:lint` Scan F.5 (LLM-judge verdicts at `confidence > 0.8`). | Evidence Trail append + info-level CLARIFICATION with LLM reasoning; owner can override. |
| **L3 — Human-gated via CLARIFICATION** | Land a new principle; deprecate an existing one; change `priority_tier`; change `core` flag; change `scope` / `applies_to`; rewrite `statement`; confirm `last_reviewed`. | Owner manually, as part of CLARIFICATIONS sweep. | Weekly / monthly batch review. |
| **L4 — Never automatic** | Change content inside any `core: true` note; promote a principle to `priority_tier: 1`; assign `scope: sensitive`. | Owner only, with deliberate double-read before save. | Manual edit. Even in batch review, pause. |

### 8.1 Safety invariants

- Automation **never** edits the body of a `core: true` note (apart from appending
  to its Evidence Trail at L1).
- Automation **never** creates a new `axiom`. Candidates that look like axioms are
  surfaced via CLARIFICATION; only owner lands them.
- Auto-merge (L2) requires matching `scope` (never merges `shared` into `personal`
  or vice versa) and matching `type`.
- `status: placeholder` notes are treated as synthetic test data. They are not
  emitted in shareable exports, not included in `constitution-core.md` even if
  they have `core: true`, and are replaced during content population.

## 9. Evidence Trail Lifecycle

The `## Evidence Trail` section is the **single audit log** for each principle.
No parallel log files live inside `0_constitution/`. Everything that happens to
the principle appears as a dated entry in this section.

**Format** (same as Evidence Trail in knowledge notes):

```
- **YYYY-MM-DD** | {event-type} | [[link-or-ref]] — {one-line description}
```

**Event types:**

| Event | Emitter | Example body |
|---|---|---|
| `extraction` | `/ztn:bootstrap --layer constitution` or manual authoring | `extraction from [[source-note]] — tagged as axiom candidate` |
| `landing` | Owner manual | `landed in constitution after human review` |
| `refinement` | Owner manual or session edit | `rephrased in session [[session-id]] — {reason}` |
| `citation-aligned` | `/check-decision` auto-append | `cited in [[record]] — verdict: aligned` |
| `citation-violated` | `/check-decision` auto-append | `contradicted by [[record]] — deviation: {reason}` |
| `citation-tradeoff` | `/check-decision` auto-append | `tradeoff in [[record]] — with [[other-principle-id]]` |
| `stale-surfaced` | `/ztn:lint` Scan F.1 | `surfaced as stale (>180 days since last_reviewed)` |
| `review-confirmed` | Owner manual | `confirmed on review — still applicable` |
| `deprecated` | Owner manual | `deprecated — reason: {reason}; status: archived` |
| `automerge-exact` | `/ztn:lint` Scan F.5 | `absorbed exact-duplicate candidate [[candidate-ref]]` |
| `automerge-llm` | `/ztn:lint` Scan F.5 | `absorbed semantic duplicate candidate [[candidate-ref]] — LLM confidence: 0.{NN}` |
| `compacted` | `compact_evidence_trail.py` | `[compacted] YYYY-MM..YYYY-MM — {N} entries → summary: {one-line}` |

**Side effect of citations:**
- `citation-aligned` / `citation-violated` / `citation-tradeoff` all bump
  `last_applied: today` in the frontmatter (L1 automation).
- Length of Evidence Trail is a believability signal: richly-cited principles
  weigh more in tie-breaking than aspirational ones.

**Compaction** (triggered by `/ztn:lint` Scan F.7 when entries > 50):
- LLM proposes a one-line summary for entries older than 6 months.
- Summary enters CLARIFICATION; owner approves / rejects / edits options.
- On approval, `compact_evidence_trail.py` rewrites the section between sentinel
  markers. Compacted entries keep a `[compacted]` prefix. The most recent year is
  never compacted. Original pre-compaction file remains in git history.

## 10. CLARIFICATION Types

Constitution emits the following CLARIFICATION types into
`_system/state/CLARIFICATIONS.md`. All are typed for grep + tooling; all reach owner via
the same existing review flow (no new cadence).

| Type | Emitted by | Trigger | Required human action |
|---|---|---|---|
| `principle-stale` | `/ztn:lint` Scan F.1 (daily) | `last_reviewed` > 180 days | Confirm / rephrase / deprecate |
| `principle-drift` | `/ztn:process` Step 3.x | Decision contradicts active principle, `confidence ≥ 0.8` | Was deviation intentional? Refine principle, add exception, or reconsider decision |
| `principle-drift-retro` | `/ztn:lint --rescan-drift` (manual) | Historical drift found after principle edit | Same as above; scoped to retrospective decisions |
| `principle-tradeoff` | `/ztn:process` Step 3.x | Decision trades two principles | Confirm the trade-off; optionally log explicit reasoning into the record |
| `principle-candidate-batch` | `/ztn:lint` Scan F.3 (weekly) | Candidates buffer non-empty | Review candidates inline: promote / reject / defer |
| `principle-automerge-exact` | `/ztn:lint` Scan F.5 (daily, info) | Exact content match between candidate and active principle | Info-only; owner may override if absorption was wrong |
| `principle-automerge-llm` | `/ztn:lint` Scan F.5 (daily, info) | LLM-judge verdict `(a)` with `confidence > 0.8` | Info-only; LLM reasoning preserved for audit; override possible |
| `principle-extended-llm` | `/ztn:lint` Scan F.5 (daily, info) | LLM-judge verdict `(b)` with `confidence > 0.8` | Info-only; candidate added as edge-case reference; override possible |
| `core-bloat` | `/ztn:lint` Scan F.6 (daily) | `core: true` count > 10 | Revisit compression — which core entries can be demoted or derived? |
| `evidence-trail-compact` | `/ztn:lint` Scan F.7 (weekly) | Principle Evidence Trail > 50 entries | Choose: LLM-compact / keep / selective |
| `lint-archive-failure` | `/ztn:lint` Scan F.3 (weekly) | Candidate archive write + read-back failed | Investigate; buffer was NOT cleared (candidates preserved) |
| `soul-manual-edit-to-auto-zone` | `render_soul_values.py` pre-commit | Text between SOUL auto-zone markers differs from expected render without matching `0_constitution/` change | Owner intended to edit a principle but edited the view |
| `principle-automerge-scope-mismatch` | `/ztn:lint` Scan F.5 (daily) | Candidate matches principle semantically but scopes differ | Owner decides which scope is correct before merge |

See [`_system/state/CLARIFICATIONS.md`](../_system/state/CLARIFICATIONS.md) for how types are
formatted and resolved in-line (this registry is the canonical type list).

## 11. Evolution Ladder

How an observation becomes part of the core.

```
 (1) Observation in session
      ↓  [/ztn:capture-candidate — L1, auto]
 (2) Candidate in _system/state/principle-candidates.jsonl
      ↓  [/ztn:lint F.5 daily: either auto-merge/extend at L2, or wait]
 (3a) Merged into existing principle (Evidence Trail appended)
       ↓  [Evidence Trail grows — L1, auto]
 (3b) Aggregated into weekly F.3 CLARIFICATION
       ↓  [human review — L3, owner promote/reject/defer]
 (4a) Rejected → archived in
      _system/state/lint-context/weekly/{YYYY-WW}-candidates-archived.jsonl
      (never silent delete; full audit trail forever)
 (4b) Deferred → remains in buffer with explicit deferred marker
 (4c) Promoted → new note in 0_constitution/{type}/{domain}/
       ↓  [time passes; Evidence Trail grows through real citations]
 (5) Tier elevation / core promotion — ALWAYS manual (L3/L4)
      - F.4 monthly surfaces metrics: most-cited, most-stable, most-contradicted.
      - Metrics are input to owner's thinking; never an auto-trigger.
      - Promotion to core = L4; always owner, never automation.
 (6) Stale review — every 180 days via F.1 → confirm / deprecate / demote
 (7) Deprecated → status: archived; file stays in tree; lives in git history
```

**Never silent, ever.** Every transition produces either a CLARIFICATION, an
Evidence Trail entry, or an archive file. Auditable end-to-end.

## 12. How to Use in Sessions

### 12.1 As a Claude agent

- The core (`~/.claude/rules/constitution-core.md`) is always loaded. For any
  non-trivial decision that seems to touch constitution domains (ethics, identity,
  work philosophy, relationships) — invoke `/check-decision` before acting.
- When you observe one of the capture triggers (see `~/.claude/CLAUDE.md`
  "Constitution capture" section) — invoke `/ztn:capture-candidate`. Do not ask
  permission. The buffer is append-only; owner resolves in batch.
- Do not edit any file under `0_constitution/` directly. The only allowed writes
  are L1 actions through the scripts / skills listed in the matrix above.

### 12.2 As Owner

- **Authoring** a new principle: write the `.md` by hand, include full frontmatter,
  seed the Evidence Trail with at least one `extraction` or `landing` entry.
  Run `/ztn:regen-constitution` to refresh derived views, then `git add`.
- **Refining** an existing principle: edit the file; run
  `/ztn:regen-constitution`; add a `refinement` Evidence Trail entry.
- **Deprecating** a principle: set `status: archived`, append a `deprecated`
  Evidence Trail entry of the form `deprecated — reason: {reason}; status: archived`.
  The `reason: {reason}` payload is the contract-required archival reason
  (Archive Contract Form A constitution-principle exception in
  `_system/docs/SYSTEM_CONFIG.md`) — do NOT also append a separate
  `## Archive Note` block. Do not delete the file.
- **Reviewing** CLARIFICATIONS: normal `CLARIFICATIONS.md` sweep. Constitution
  items mix with all other clarifications; resolve them the same way.

### 12.3 As a consumer outside Claude Code

- External AIs (ChatGPT, Claude Desktop public tier) receive only `shared`
  content through the MCP public token (when MCP is available).
- Shareable snapshots for external users are not in the current surface set.
  When that work lands, it builds on the existing `scope` field — no schema
  change required.

## 13. Governance Invariants (HARD RULES)

These are non-negotiable. A trade-off against any of them must be surfaced and
approved by owner before any code is written.

1. **No silent compromises.** Any trade-off — performance vs clarity, correctness
   vs effort, duplication vs unification, safety vs speed — is surfaced to owner
   before the choice is made. Applies to this constitution system as much as to
   any code built against it.

2. **Human-in-the-loop on content and governance metadata.** L3 and L4 actions
   (see §8) never occur without owner. Automation is restricted to L1 and
   narrowly-authorised L2.

3. **Source of truth in ZTN.** The canonical form of every principle is the
   markdown file under `0_constitution/`. Every other surface is a derived view.
   Editing a view is wasted effort — it regenerates from source on next run.

4. **No new cadences.** Constitution maintenance piggy-backs on the existing
   `/ztn:lint` nightly and the existing CLARIFICATIONS sweep. No new review
   ritual, no new cron entries beyond the nightly lint.

5. **Stale is dangerous.** `last_reviewed > 180 days` surfaces a CLARIFICATION.
   Alive or dead — not a zombie.

6. **Positive + behavioral framing by default.** "Choose / Do / Avoid X when Y",
   not trait descriptors. Empirically best for AI compliance (C3AI).

7. **Optional for agents.** Small core always on; full tree on demand via
   `/check-decision` or MCP. Agents should not be forced to load the entire tree
   to answer a trivial question.

8. **No premature abstraction.** Three types, no separate `decision-log/` folder
   (use `_records/` with `types: [decision]`), no separate `candidates/` folder
   (use `status: candidate` in frontmatter), no `CONFLICTS.md` registry.

9. **Resistance to model churn.** Constitution lives in files, not in model
   weights. Migrating to a new model changes nothing in this tree.

10. **Scope is filter-on-consumer, not state-on-principle.** A principle's scope
    is fixed; consumers filter. Same content, different views per consumer role.

11. **Single audit log per principle.** The `## Evidence Trail` section is the
    only history kept inside a constitution note. Lint / process / maintain may
    write to it at L1; they do not create parallel log files inside
    `0_constitution/`.

12. **Write without commit.** All automated updates (Evidence Trail, `last_applied`,
    regenerated index/core/SOUL.Values) are written into the dirty tree. Commits
    are always made by owner.

13. **Compression discipline — the core is irreducible.** Target: 5–8 notes with
    `core: true` across the whole constitution. More than 10 surfaces a
    `core-bloat` CLARIFICATION. Bloat defeats the purpose.

14. **Transformation scripts use no LLM.** Mechanical work (filtering, rendering,
    indexing, archiving) is Python + PyYAML, deterministic. LLM use is reserved
    for reasoning tasks (`/check-decision`, `/ztn:lint` Scan F.5 merge judgment,
    `/ztn:lint` Scan F.7 compaction suggestion).

15. **Capture agent writes only to the buffer.** `/ztn:capture-candidate` never
    decides anything about the constitution. It appends to
    `_system/state/principle-candidates.jsonl`. Compression discipline is not spread
    across many agents.

16. **Never silent delete.** Rejected candidates move to the weekly archive file
    with verify-before-clear. Archive failure blocks buffer clearing and raises
    a `lint-archive-failure` CLARIFICATION.

17. **LLM-judge for semantic merge, not embedding similarity.** Exact content
    match is the only automation without LLM. Everything else routes through
    an Opus-backed verdict with explicit JSON `{verdict, target_principle_id,
    confidence, reasoning}`. Reasoning is preserved in the CLARIFICATION for
    audit.

18. **Placeholder status is test data.** Notes with `status: placeholder` exist
    only as transient fixtures for smoke-testing the infrastructure. They are
    excluded from exports, from `constitution-core.md`, and from SOUL.Values
    regardless of any other flag. Placeholders MUST NOT appear in the active
    content set — they are allowed only as short-lived fixtures during infra
    changes and must be removed before the change is merged.

# If it can be better, it should be better

## Statement
When the system I am building is likely to outlive the current decision — same

## Why I hold this
Every shortcut becomes a tax later, paid by someone who didn't choose it —
future me, my team, my family. The decisions that compound positively over
time share one pattern: choose the harder path today because the cost of
revisit is much higher than the cost of correct setup now.

  is truly throwaway.)
- Decisions whose reversal cost is trivial and whose value-of-information
  outweighs the quality gap — speed-to-learning wins.

## Appendix B — Checklist for authoring a new principle

1. Does it fit one of three types? If not, stop — it is probably knowledge,
   a preference, or a team convention.
2. Does it fit one existing domain? If two — split or promote to `meta`.
3. Is the statement behavioral and positive by default?
4. Is `scope` set deliberately? (Default to `personal` unless you genuinely
   want to share it.)
5. Does `applies_to` list match reality — will each listed consumer actually
   use this?
6. Does `priority_tier` match the domain default, or is there a reason to
   deviate (documented in the body)?
7. Is `core: true` justified? Target 5–8 total. If unsure — leave as `false`.
8. Is the first Evidence Trail entry present (`extraction` or `landing`)?
9. Is `confidence` honest? (`experimental` is fine — aspirational is not.)
10. `git add` only. Do not `git commit` — owner runs the commit.
